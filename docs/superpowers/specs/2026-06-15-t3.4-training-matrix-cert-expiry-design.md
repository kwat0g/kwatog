# T3.4 — Training Matrix + Certification Expiry Automation — Design

**Track 3, Task 4.** Greenfield surface for tracking employee training + certification records, with automatic expiry alerts that fire once per threshold (30/14/7 days out, then on/after the expiry date itself). Closes the IATF 16949 training-records gap that auditors check during surveillance audits.

## Status quo

The HR module today tracks employees, departments, positions, employment history, and documents — but not training or certification records. There is no:

- Catalog of training programs or certification types.
- Per-employee record of completed / scheduled trainings.
- Visibility into upcoming certificate expiries.
- Notification to HR / department heads when a forklift licence (etc.) is about to lapse.

For an IATF-certified plant the gap is real: operators with expired hot-work permits, expired forklift licences, expired food-safety cards, etc. are non-conformances waiting to be found.

## Scope (3 sub-features)

| ID | Feature | One-liner |
|----|---------|-----------|
| T3.4.A | Training catalog + employee training records | `trainings` + `employee_trainings` tables with full CRUD + assign/complete service. |
| T3.4.B | Auto-compute `expires_at` on completion | When `completed_at` is set, derive `expires_at = completed_at + validity_months` (when validity_months is present on the catalog row). |
| T3.4.C | Expiry alert cron with tiered thresholds | Daily `training:check-expiries` command — scan upcoming expiries at 30 / 14 / 7 days out and on-or-after expiry. Each threshold fires once per record via `last_alert_level`. |

Out of scope (deferred):

- Position → required-trainings matrix (Track 4 or T3.5).
- Self-service training enrollment or e-learning integration.
- Cert file upload via the new endpoints — the existing HR document surface already handles file uploads, employee can attach a certificate manually.
- Skill-level grading (basic / intermediate / expert) — binary completed/not is enough for thesis demo.
- Trainer assignment, room booking, attendance roster — the new service records that a training was completed, not how it was delivered.

## Architecture

```
trainings (catalog)
  ├─ id, name, description, duration_hours, validity_months (nullable),
  │  is_certification (bool), is_active
  └─ optional: department_id (some trainings are dept-scoped)

employee_trainings (records)
  ├─ id, employee_id FK, training_id FK,
  │  scheduled_for date NULLable,
  │  completed_at date NULLable,
  │  expires_at date NULLable,
  │  status enum (scheduled | completed | expired | cancelled),
  │  certificate_path nullable string,
  │  notes,
  │  last_alert_level enum nullable (t30 | t14 | t7 | expired),
```

Three new controllers + permission slugs:

- `EmployeeTrainingController` (admin assign/record/list) — `hr.employees.trainings.manage` + `hr.employees.trainings.view`

## T3.4.A — Catalog + records

`Training` model with `HasHashId`, `HasAuditLog`. Soft delete optional — keep simple, don't add SoftDeletes for now (auditors can read deleted rows from `audit_logs` if needed).

`EmployeeTraining` model with `HasHashId`, `HasAuditLog`, `belongsTo Employee` and `belongsTo Training`.

Service:

```php
class EmployeeTrainingService
{
    public function assign(
        Employee $employee,
        Training $training,
        ?Carbon $scheduledFor = null,
        ?User $by = null,
    ): EmployeeTraining;

    public function recordCompletion(
        EmployeeTraining $record,
        Carbon $completedAt,
        ?string $certificatePath = null,
        ?User $by = null,
    ): EmployeeTraining;

    public function cancel(EmployeeTraining $record, ?string $reason = null, ?User $by = null): EmployeeTraining;
```

## T3.4.B — Auto-compute `expires_at`

Inside `EmployeeTrainingService::recordCompletion()`:

```php
$training = $record->training;  // eager-loaded
$expiresAt = $training->validity_months
    ? Carbon::parse($completedAt)->addMonths((int) $training->validity_months)
    : null;

$record->forceFill([
    'completed_at'     => $completedAt,
    'expires_at'       => $expiresAt,
    'status'           => 'completed',
    'certificate_path' => $certificatePath ?? $record->certificate_path,
    'last_alert_level' => null,                  // reset alerts on re-completion
    'last_alert_at'    => null,
])->save();
```

Validity months can be null (e.g. one-off orientation training) — those records simply have no `expires_at` and never enter the expiry cron.

## T3.4.C — Expiry alert cron

```php
class TrainingExpiryService
{
    private const TIERS = [
        // [days-out threshold, level token, message tone]
        ['days' => 30, 'level' => 't30',     'tone' => 'reminder'],
        ['days' => 14, 'level' => 't14',     'tone' => 'soon'],
        ['days' => 7,  'level' => 't7',      'tone' => 'urgent'],
        ['days' => 0,  'level' => 'expired', 'tone' => 'overdue'],
    ];

    public function check(): array  // returns ['evaluated' => N, 'alerts_sent' => M, 'expired_marked' => K]
    {
        $today = now()->startOfDay();

        $rows = EmployeeTraining::query()
            ->with(['employee.department', 'training'])
            ->where('status', 'completed')
            ->whereNotNull('expires_at')
            ->where('expires_at', '<=', $today->copy()->addDays(30))
            ->get();

        $alertsSent = 0;
        $expiredMarked = 0;

        foreach ($rows as $r) {
            // Determine the most-severe tier this row is past.
            $daysUntil = $today->diffInDays(Carbon::parse($r->expires_at), false);

            $tier = null;
            foreach (self::TIERS as $t) {
                if ($daysUntil <= $t['days']) {
                    $tier = $t;
                }
            }
            if (! $tier) continue;

            // Fire once per tier — last_alert_level upgrades only forward.
            if ($this->alreadyFired($r, $tier['level'])) continue;

            $this->notify($r, $tier);

            $r->forceFill([
                'last_alert_level' => $tier['level'],
                'last_alert_at'    => now(),
                'status'           => $tier['level'] === 'expired' ? 'expired' : $r->status,
            ])->save();

            $alertsSent++;
            if ($tier['level'] === 'expired') $expiredMarked++;
        }

        return [
            'evaluated'      => $rows->count(),
            'alerts_sent'    => $alertsSent,
            'expired_marked' => $expiredMarked,
        ];
    }
}
```

Tier ordering is: `t30 → t14 → t7 → expired`. `alreadyFired($row, $level)` short-circuits when `last_alert_level` has reached or surpassed the tier (so re-runs the same day don't re-fire).

Notification recipients per fire: the employee (own `User` row, via `users.employee_id`), the employee's department head (`role_slug = department_head` AND same department), and all users with role `hr_officer`.

Console:

```php
class CheckTrainingExpiries extends Command
{
    protected $signature = 'training:check-expiries';
    protected $description = 'Send tiered alerts for expiring/expired training records';

    public function handle(TrainingExpiryService $svc): int
    {
        $r = $svc->check();
        $this->info(sprintf(
            'Training expiry check: %d evaluated, %d alerts, %d expired.',
            $r['evaluated'], $r['alerts_sent'], $r['expired_marked'],
        ));
        return self::SUCCESS;
    }
}
```

Schedule registration in `routes/console.php`:

```php
// T3.4.C — Daily training expiry alerts.
Schedule::command('training:check-expiries')
    ->dailyAt('06:30')
    ->withoutOverlapping()
    ->onOneServer();
```

## Acceptance criteria

1. Catalog CRUD endpoints work and are permission-gated.
2. Assigning a training creates a row with `status='scheduled'`. Re-assigning the same employee to the same `(training_id, scheduled_for)` raises a 422 "already assigned" via the unique index.
3. Completing a training stamps `completed_at`, derives `expires_at` from the catalog's `validity_months` when present, and flips status to `completed`. Re-completion (rare, e.g. retake) resets `last_alert_level` so alerts can re-fire on the new expiry.
4. `php artisan training:check-expiries` fires t30/t14/t7/expired tiers exactly once per record. Status flips to `expired` on the on-or-after-expiry tier.
5. Re-running the cron the same day is a no-op for already-fired tiers.
6. Notifications go to: employee (if a `users` row exists for the employee), their dept head, and all hr_officers.
7. Each mutation produces exactly one `audit_logs` row per logical action.
8. Full suite green: 687 + 18 = 705 passing.
9. No SPA work — admin can manage via `php artisan tinker` for thesis demo. SPA is T3.4-spa (deferred).

## Out-of-scope (explicit)

- Self-service "I attended this training, please record it" flow — admin records on behalf for now.
- Bulk import of training history from spreadsheet — manual one-by-one for thesis demo.
- Position → required-trainings matrix.
- Reminders BEFORE the t30 tier (e.g. 60 days, 90 days). Fixed thresholds keep the surface small.
- Automatic suspension of an employee with an expired licence — admin sees the expired record and decides.
- Customer / supplier training history — only employees.
- SPA UI surface for the new endpoints (deferred to a follow-up ticket; admin uses tinker for demo).
