# T3.1 — NCR Follow-Through Automation — Design

**Track 3, Task 1.** Closes the gap between an NCR being raised and IATF 16949
demanding evidence that the corrective + preventive actions actually happened,
were verified, and didn't recur. Today's NCR loop opens cleanly but lets
non-conformances drift without owner, due date, rework execution, or
recurrence visibility. T3.1 wires automation around the existing NCR
surface — no rewrites.

## Status quo (audited)

Already in place:

- `InspectionService::recordMeasurements()` auto-opens an NCR via
  `NcrService::openFromInspectionFailure()` on inspection fail (line 298+).
- `NcrService::create / addAction / setDisposition / close / cancel`.
- `NcrAction` model — Containment / Corrective / Preventive types.
- Disposition enum — Scrap / Rework / UseAsIs / ReturnToSupplier.
- On `close()`: outgoing-QC + Scrap → auto-replacement WO with
  `parent_ncr_id`. ReturnToSupplier → notify Purchasing.
- QC Head notification on auto-NCR creation.

Confirmed gaps (this design fills them):

1. `close()` does **not** require a Corrective action row to exist —
   IATF auditors fail on "NCR closed without documented corrective".
2. No SLA escalation: an open NCR with severity=High can sit untouched
   for weeks. No escalator email.
3. Disposition=Rework has **no** auto rework-execution work item.
   Only Scrap auto-creates a replacement WO.
4. No recurrence detection — same product + same defect signature
   reappearing within a window doesn't flag systemic CA.
5. `NcrAction` has no `due_date` or `verified_at` — effectiveness
   verification (IATF 8D step D7) is invisible.

## Scope (4 sub-features)

| ID | Feature | One-liner |
|----|---------|-----------|
| T3.1.A | Mandatory CA + PA on close | `close()` rejects unless ≥1 Corrective AND ≥1 Preventive `NcrAction` row exists. |
| T3.1.B | Auto rework WO on Rework disposition | Mirror the Scrap → replacement WO path: Rework + outgoing-QC → auto-create rework WO. |
| T3.1.C | NCR SLA escalation cron | Open NCRs with severity High/Critical and no Corrective action within SLA → escalation notification ladder. |
| T3.1.D | Recurrence detection | On NCR create, scan last 30 days for same `product_id` + same defect signature; tag and notify. |

Out of scope (deferred):

- Structured 5 Whys / 8D timeline UI — T3.2.
- Effectiveness re-check workflow with re-inspection scheduling — T3.2.
- Custom SLA per severity per customer — fixed defaults are fine for thesis.
- COPQ rollup — T3.6.
- Bulk close — manual case-by-case is correct for IATF.

## Architecture

T3.1 reuses three patterns the codebase already proved:

- Track 1's per-row try/catch bulk pattern (none of T3.1 is bulk, but the
  escalation cron iterates with the same defensive shape).
- Track 1's approval SLA escalator (`ApprovalEscalationService` + console
  cmd + scheduler) — T3.1.C clones it for NCRs.
- Existing `NcrAutoRaised` / `NotificationService` plumbing — T3.1.D
  reuses for recurrence flag.

Five new pieces, all small:

```
T3.1.A  NcrService::close()                  — guard added in same method
T3.1.B  NcrService::close()                  — branch added next to existing Scrap
T3.1.C  api/app/Modules/Quality/Services/NcrEscalationService.php   (new)

        Console\Kernel::schedule()           — append entry
T3.1.D  api/app/Modules/Quality/Services/NcrRecurrenceDetector.php  (new)
        Hook from NcrService::create() afterCommit
```

No SPA changes; all backend.

## Data model changes

Single migration. Additive, all nullable, no backfill required:

```sql
ALTER TABLE ncr_actions
  ADD COLUMN due_date date NULL,
  ADD COLUMN owner_id bigint NULL REFERENCES users(id) ON DELETE SET NULL,
  ADD COLUMN verified_at timestamp NULL,
  ADD COLUMN verified_by bigint NULL REFERENCES users(id) ON DELETE SET NULL;

ALTER TABLE non_conformance_reports
  ADD COLUMN escalation_level smallint NOT NULL DEFAULT 0,
  ADD COLUMN last_escalated_at timestamp NULL,
  ADD COLUMN recurrence_of_ncr_id bigint NULL REFERENCES non_conformance_reports(id) ON DELETE SET NULL,
  ADD COLUMN rework_work_order_id bigint NULL REFERENCES work_orders(id) ON DELETE SET NULL;

CREATE INDEX ix_ncr_open_severity ON non_conformance_reports (status, severity, last_escalated_at);
CREATE INDEX ix_ncr_recurrence ON non_conformance_reports (product_id, created_at);
```

`due_date`/`owner_id` etc. on `NcrAction` give T3.2 something to build the
8D timeline against without another migration. They're nullable so legacy
rows keep working.

## T3.1.A — Mandatory CA + PA on close

`NcrService::close()` adds two pre-checks before the status flip:

```php
$counts = $ncr->actions()
    ->selectRaw('action_type, COUNT(*) as c')
    ->groupBy('action_type')
    ->pluck('c', 'action_type');

if (($counts[NcrActionType::Corrective->value] ?? 0) < 1) {
    throw new RuntimeException('Cannot close NCR without at least one Corrective action.');
}
if (($counts[NcrActionType::Preventive->value] ?? 0) < 1) {
    throw new RuntimeException('Cannot close NCR without at least one Preventive action.');
}
```

The existing `cancel()` path is unchanged — cancelled NCRs do not require
actions, since cancellation means "this NCR was raised in error".

The two existing tests that close NCRs (`InspectionNcrTest` + Disposition
tests) already add actions before closing; if any test closes without
actions it gets updated as part of T3.1.A's task list.

## T3.1.B — Auto rework WO on Rework disposition

Replays the Scrap branch's shape:

```php
if ($ncr->disposition === NcrDisposition::Rework
    && $ncr->inspection_id
    && $ncr->product_id
    && $ncr->affected_quantity > 0) {
    $insp = Inspection::find($ncr->inspection_id);
    if ($insp && $insp->stage === InspectionStage::Outgoing) {
        $wo = $this->workOrderService()?->createDraft([
            'product_id'      => $ncr->product_id,
            'quantity_target' => $ncr->affected_quantity,
            'planned_start'   => now()->addDay()->toDateString(),
            'planned_end'     => now()->addDays(3)->toDateString(),  // tighter than Scrap (3d vs 7d)
            'priority'        => 7,                                   // higher than Scrap-replacement
            'parent_ncr_id'   => $ncr->id,
            'created_by'      => $by->id,
            'wo_type'         => 'rework',                            // new optional field on WO if it doesn't exist; otherwise omitted
        ]);
        if ($wo) {
            $ncr->forceFill(['rework_work_order_id' => $wo->id])->save();
        }
    }
}
```

If `WorkOrder` doesn't have a `wo_type` column today, the implementer
drops the line — naming the WO `REWORK-…` via the existing prefix is
sufficient demo signal. (Plan task verifies before coding.)

## T3.1.C — NCR SLA escalation

Mirrors `ApprovalEscalationService` from Track 1.

Default ladder (constants on the service, not in DB — keeps it explicit):

```php
private const SLA_HOURS_BY_SEVERITY = [
    'critical' => 8,    // critical NCR with no corrective → 8h to escalate
    'high'     => 24,
    'medium'   => 72,
    'low'      => 168,
];
private const ESCALATION_TIERS = [
    1 => ['role' => 'qc_inspector',   'subject' => 'NCR awaiting corrective'],
    2 => ['role' => 'plant_manager',  'subject' => 'NCR overdue — manager attention'],
    3 => ['role' => 'system_admin',   'subject' => 'NCR critical overdue — exec escalation'],
];
```

`NcrEscalationService::run()`:

1. `NonConformanceReport::query()->where('status', Open)->whereDoesntHave('actions', fn ($q) => $q->where('action_type', Corrective))` — open NCRs without a corrective.
2. For each, compute hours since `created_at` minus `escalation_level * sla_hours[severity]` (each tier resets the clock once notified).
3. If due, increment `escalation_level`, stamp `last_escalated_at`, dispatch notification to all users with the tier's role.
4. Cap at tier 3.

Console command: `php artisan ncr:escalate` (idempotent — if not due, no-op).
Schedule: every 15 minutes (matches approval escalator cadence).

Tests cover: not-yet-due ⇒ no email; just-due ⇒ tier 1 email; tier 1 already
sent + 24h passed ⇒ tier 2 email; tier 3 cap.

## T3.1.D — Recurrence detection

On `NcrService::create()` afterCommit, run `NcrRecurrenceDetector::scan($ncr)`:

1. Define defect signature = first non-empty of: failed measurement's
   `parameter_name`, otherwise hash of `defect_description` first 80 chars
   (lowercased + whitespace-collapsed).
2. Query: NCRs in last 30 days with same `product_id` and same signature,
   excluding the new row.
3. If ≥1 prior, set `recurrence_of_ncr_id` to the most recent prior, then
   notify `qc_inspector` + `plant_manager` roles with subject
   "Recurring NCR: {product} — {n} occurrences in 30 days".

`scan()` must NOT throw — wrap in try/catch + log. A recurrence-detector
failure cannot block NCR creation.

Idempotency: re-running scan on the same NCR sees that
`recurrence_of_ncr_id` is already set and short-circuits.

## Acceptance criteria

1. `NcrService::close()` rejects with 422 if missing Corrective or
   Preventive action.
2. NCR with disposition=Rework on outgoing-QC inspection auto-creates
   a rework WO with `parent_ncr_id` and `priority=7`.
3. `php artisan ncr:escalate` advances tier exactly once per due window
   and stops at tier 3.
4. Creating an NCR for product P with the same defect signature within
   30 days of a prior NCR sets `recurrence_of_ncr_id` and notifies
   QC + plant manager.
5. All sub-features are observable in audit_logs (single audit row per
   logical action — Track 1 cleanup #16 pattern).
6. Full suite green: 649 + 17 = 666 passing.
7. No SPA work; backend only.

## Risks + mitigations

- **Migration on `non_conformance_reports`** in a hot table — all
  additive, all nullable, no backfill, indexed columns are new. Safe.
- **Escalation noise** — fixed defaults (8/24/72/168h) chosen
  conservatively; if production tunes them later, lift to settings table
  in T3.6.
- **Recurrence false positives** — defect signature hashing is naive on
  free-text descriptions. Acceptable for thesis; can refine in T3.2.
- **Backfill** — existing open NCRs get `escalation_level=0` so they
  enter the ladder fresh. No retroactive emails.
