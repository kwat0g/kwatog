# Track 1, Task 6: Approval SLA Auto-Resolve — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use `- [ ]` syntax.

**Goal:** Today the approval escalation engine only nags. Stale approvals rot in the queue forever. This task adds auto-resolve: after a record has been escalated and stays untouched for `auto_resolve_after_hours`, the system either auto-approves or auto-rejects (per workflow policy), records who and when, and lets the chain advance. Eliminates the human "chase the VP" task.

**Architecture:** Extend the existing `ApprovalEscalationService` with `runAutoResolve()`. Iterate `approval_records` that are `pending`, `escalated_at IS NOT NULL`, `auto_resolved_at IS NULL`, and where `now() - escalated_at >= auto_resolve_after_hours`. The action (`approve` / `reject`) is decided by, in priority order: (1) the `auto_resolve_action` field on the matching workflow step JSON, (2) the global setting `approvals.auto_resolve.default_action`. The number of hours to wait is decided by, in priority order: (1) `auto_resolve_after_hours` on the step, (2) global setting `approvals.auto_resolve.default_hours`. The action mutates the row exactly like a human-driven `approve/reject` (cascade-skips later steps on reject) plus stamps `auto_resolved_at`. A synthetic system user (first `system_admin`) is recorded as `approver_id` so audit logs and downstream listeners have a real user FK. Wired into the existing `approvals:run-escalations` command — no new schedule.

**Tech Stack:** Laravel 11, Postgres 16, existing approval engine. No new mailable, no SPA changes.

## Background

- `ApprovalRecord` columns (mig 0010 + 0115): `approvable_type`, `approvable_id`, `step_order`, `role_slug`, `approver_id`, `action`, `remarks`, `acted_at`, `created_at`, `reminder_sent_at`, `escalated_at`, `escalated_to_user_id`. Action enum (string, no PHP enum): `pending` / `approved` / `rejected` / `skipped`.
- `WorkflowDefinition` `steps` is a JSON array. Each entry currently has `order`, `role`, optional `threshold`. We will read two NEW optional fields: `auto_resolve_after_hours` (int) and `auto_resolve_action` (`approve` | `reject`). Existing rows without those fields fall back to the global settings.
- Reject cascade: on rejection of step N, all later pending steps must be marked `skipped` (mirror `ApprovalService::reject`).
- Skip side-effects on the approvable model itself (status changes etc.) — the auto-resolve only mutates the `approval_records` row. The next time an approver acts (or a separate listener reads the chain), the model status will reflect the chain state. This is intentional: T1.6 is a chain-level fix, not a model-mutation fix.
- The `approvals:run-escalations` command runs every 6 hours per `routes/console.php`. Auto-resolve uses the same cadence — that is fine because the SLA is in hours not minutes.

```php
    /**
     * T1.6 — Auto-resolve approval records that have been escalated for too long.
     * Returns the count of records auto-resolved.
     */
    public function runAutoResolve(): int
    {
        if (! (bool) $this->settings->get('approvals.auto_resolve.enabled', false)) {
            return 0;
        }

        $defaultHours  = (int) $this->settings->get('approvals.auto_resolve.default_hours', self::DEFAULT_AUTO_RESOLVE_HOURS);
        $defaultAction = (string) $this->settings->get('approvals.auto_resolve.default_action', self::DEFAULT_AUTO_RESOLVE_ACTION);

        $count = 0;
        $stale = ApprovalRecord::query()
            ->where('action', 'pending')
            ->whereNotNull('escalated_at')
            ->whereNull('auto_resolved_at')
            ->get();

        foreach ($stale as $rec) {
            try {
                [$hours, $action] = $this->resolvePolicyForRecord($rec, $defaultHours, $defaultAction);
                if ($hours <= 0) continue;
                if (now()->diffInHours($rec->escalated_at, false) > -$hours) {
                    // Not stale enough yet (Carbon::diffInHours with absolute=false is signed).
                    continue;
                }
                $this->autoResolveRecord($rec, $action);
                $count++;
            } catch (\Throwable $e) {
                Log::warning('ApprovalEscalationService::autoResolve failed', [
                    'record_id' => $rec->id,
                    'error'     => $e->getMessage(),
                ]);
            }
        }
        return $count;
    }

    /**
     * Find the workflow step matching this record and read its
     * `auto_resolve_after_hours` and `auto_resolve_action`. Falls back to
     * the supplied defaults.
     *
     * @return array{0:int, 1:string} [hours, action]
     */
    private function resolvePolicyForRecord(ApprovalRecord $rec, int $defaultHours, string $defaultAction): array
    {
        $hours  = $defaultHours;
        $action = $defaultAction;

        // Resolve the workflow_type by walking back to ApprovalService::submit's
        // contract: each approval_records row carries (approvable_type, role_slug,
        // step_order). We re-derive the workflow definition by matching role_slug
        // + step_order against any WorkflowDefinition. This is best-effort —
        // multiple workflow_types may share a role slug; in that case the first
        // matching workflow's policy wins. Acceptable for thesis scope; future
        // schema can stamp workflow_definition_id on the record.
        $defs = \App\Common\Models\WorkflowDefinition::query()->get();
        foreach ($defs as $def) {
            foreach (($def->steps ?? []) as $step) {
                if ((int) ($step['order'] ?? 0) === (int) $rec->step_order
                    && (string) ($step['role'] ?? '') === (string) $rec->role_slug
                ) {
                    if (isset($step['auto_resolve_after_hours'])) {
                        $hours = (int) $step['auto_resolve_after_hours'];
                    }
                    if (isset($step['auto_resolve_action'])) {
                        $action = (string) $step['auto_resolve_action'];
                    }
                    break 2;
                }
            }
        }

        if (! in_array($action, ['approve', 'reject'], true)) {
            $action = $defaultAction;
        }
        return [$hours, $action];
    }

    private function autoResolveRecord(ApprovalRecord $rec, string $action): void
    {
        $systemUser = User::query()
            ->whereHas('role', fn ($q) => $q->where('slug', 'system_admin'))
            ->where('is_active', true)
            ->orderBy('id')
            ->first();

        \Illuminate\Support\Facades\DB::transaction(function () use ($rec, $action, $systemUser) {
            $rec->update([
                'approver_id'      => $systemUser?->id,
                'action'           => $action === 'approve' ? 'approved' : 'rejected',
                'remarks'          => 'Auto-resolved by SLA policy.',
                'acted_at'         => now(),
                'auto_resolved_at' => now(),
            ]);

            if ($action === 'reject') {
                ApprovalRecord::query()
                    ->where('approvable_type', $rec->approvable_type)
                    ->where('approvable_id', $rec->approvable_id)
                    ->where('step_order', '>', $rec->step_order)
                    ->where('action', 'pending')
                    ->update(['action' => 'skipped', 'acted_at' => now()]);
            }
        });
    }
```

## Acceptance Criteria

1. With `approvals.auto_resolve.enabled = true`, a pending record whose `escalated_at` is older than the default `72h` gets `action = 'rejected'`, `auto_resolved_at = now()`, and a system_admin `approver_id`.
2. Auto-rejection cascades: any later pending steps for the same approvable become `skipped`.
3. Workflow step JSON `auto_resolve_after_hours` / `auto_resolve_action` overrides the global defaults per step.
4. With the flag off, no records are mutated.
5. Records that haven't been escalated yet, or that were escalated less than the SLA hours ago, are left alone.
6. The `approvals:run-escalations` artisan command now reports `N auto-resolved` after running reminders + escalations.
7. New tests pass; no NEW regressions in the approval test filter.
