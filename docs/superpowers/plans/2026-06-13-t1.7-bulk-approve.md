# Track 1, Task 7: Bulk Approve on Leave + Loan — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use `- [ ]` syntax.

**Goal:** Cut down on the per-request "click each row, approve, repeat 30 times" workflow. Add bulk-approve endpoints for `LeaveRequest` (both dept + HR stages) and `EmployeeLoan`. Pattern-match the existing `OvertimeService::bulkApprove` and `PurchaseRequestService::bulkApprove` so the API and behaviour stay uniform.

**Architecture:** Add three new bulk methods + three new controller endpoints + three new routes. Each bulk method:
- Accepts an array of integer IDs (post HashID-decode), the actor `User`, and optional `remarks`.
- Iterates IDs, calls the existing single-row `approve*` method inside a per-row try/catch.
- Returns `['approved' => Model[], 'failed' => [['id' => int, 'reason' => string]]]`.
- Each row's transaction stays inside the existing single-row method — one bad row never aborts the batch.

Permissions reuse the existing single-row gates (`leave.approve_dept`, `leave.approve_hr`, `loans.approve`). HashID decoding happens in the controller before calling the service.

**Tech Stack:** Laravel 11. No migration, no model change, no SPA backend. SPA list-page bulk toolbars are deferred to a later UX pass — backend-first this time so other consumers (mobile / portal) can use the endpoints right away.

**Bills:** Excluded — current `BillController` has no approve flow (only `cancel`, `recordPayment`). 3-way-match validation gates payment, no separate approve step exists. Adding one would expand scope; defer to a later task.

## Background

- `LeaveRequestService::approveDept(LeaveRequest, User, ?string)` and `approveHR(LeaveRequest, User, ?string)` — two-stage flow, status enum `PendingDept` → `PendingHr` → `Approved`.
- `LoanService::approve(EmployeeLoan, User, ?string)` — multi-step approval via the central `ApprovalService`.
- Existing `PurchaseRequestService::bulkApprove(array $ids, User $by, ?string $remarks)` returns `['approved' => [], 'failed' => [['id' => X, 'reason' => Y]]]`. Exact shape for the new methods.
- `HashIdFilter::decode($hash, ModelClass)` returns int|null. Use the same pattern as the PR controller for ID decoding.
- Tests: `OvertimeBulkApproveTest` is the existing reference for shape — partial-success scenarios + auth assertions.

### Task A: Leave service — `bulkApproveDept`

**Files:** Modify `api/app/Modules/Leave/Services/LeaveRequestService.php`

- [ ] Step 1: Append after the existing `approveHR` method:

```php
    /**
     * T1.7 — Bulk approve LeaveRequests at the department-head stage.
     * Per-row try/catch so one bad row doesn't abort the batch.
     *
     * @param array<int, int> $ids raw integer IDs (post HashID decode)
     * @return array{approved: array<int, LeaveRequest>, failed: array<int, array{id:int, reason:string}>}
     */
    public function bulkApproveDept(array $ids, User $approver, ?string $remarks = null): array
    {
        $approved = [];
        $failed   = [];

        foreach ($ids as $id) {
            try {
                $req = LeaveRequest::query()->find($id);
                if (! $req) {
                    $failed[] = ['id' => $id, 'reason' => 'Not found.'];
                    continue;
                }
                $approved[] = $this->approveDept($req, $approver, $remarks);
            } catch (\Throwable $e) {
                $failed[] = ['id' => $id, 'reason' => $e->getMessage()];
            }
        }
        return ['approved' => $approved, 'failed' => $failed];
    }

    /**
     * T1.7 — Bulk approve LeaveRequests at the HR-officer stage.
     * Per-row try/catch so one bad row doesn't abort the batch.
     *
     * @param array<int, int> $ids
     * @return array{approved: array<int, LeaveRequest>, failed: array<int, array{id:int, reason:string}>}
     */

    {
        $approved = [];
```

```php
    /**
     * T1.7 — Bulk approve dept stage.
     * Body: { ids: ["hashId1", ...], remarks?: string }
     */
    public function bulkApproveDept(Request $request): JsonResponse
    {
        $validated = Validator::make($request->all(), [
            'ids'     => 'required|array|min:1',
            'ids.*'   => 'string',
            'remarks' => 'nullable|string|max:500',
        ])->validate();

        $ids = array_filter(array_map(
            fn ($hash) => HashIdFilter::decode($hash, LeaveRequestModel::class),
            $validated['ids'],
        ));
        if (empty($ids)) {
            return response()->json(['message' => 'No valid leave request IDs provided.'], 422);
        }

        $results = $this->service->bulkApproveDept($ids, $request->user(), $validated['remarks'] ?? null);

        return response()->json([
            'data' => [
                'approved' => array_map(fn ($r) => (new \App\Modules\Leave\Resources\LeaveRequestResource($r))->toArray($request), $results['approved']),
                'failed'   => $results['failed'],

        ]);
    }
```

### Task G: Leave bulk-approve test

```php
<?php

declare(strict_types=1);

namespace Tests\Feature\Leave;

use App\Modules\HR\Models\Employee;
use App\Modules\Leave\Enums\LeaveRequestStatus;
use App\Modules\Leave\Models\LeaveRequest;
use App\Modules\Leave\Services\LeaveRequestService;
use App\Modules\Auth\Models\Role;
use App\Modules\Auth\Models\User;
use Illuminate\Foundation\Testing\RefreshDatabase;
use Tests\TestCase;

class LeaveRequestBulkApproveTest extends TestCase
{
    use RefreshDatabase;

    public function test_bulk_approve_dept_partial_success(): void
    {
        $deptHeadRole = Role::firstOrCreate(['slug' => 'department_head'], ['name' => 'Department Head']);
        $approver = User::factory()->create(['role_id' => $deptHeadRole->id, 'is_active' => true]);

        $emp = Employee::factory()->create();
        $r1 = LeaveRequest::factory()->create([
            'employee_id' => $emp->id,
            'status'      => LeaveRequestStatus::PendingDept->value,
        ]);
        // Wrong-state row → should fail (already approved at dept stage).
        $r2 = LeaveRequest::factory()->create([
            'employee_id' => $emp->id,
            'status'      => LeaveRequestStatus::PendingHr->value,
        ]);

        $svc = app(LeaveRequestService::class);
        $result = $svc->bulkApproveDept([$r1->id, $r2->id, 99999], $approver, 'batch ok');

        $this->assertCount(1, $result['approved']);
        $this->assertCount(2, $result['failed']);
        $this->assertSame(LeaveRequestStatus::PendingHr->value, $r1->fresh()->status->value);
    }
}
```

NOTE: if `LeaveRequest::factory()` requires extra fields (leave_type_id, start_date, end_date, days_requested), peek at `api/database/factories/Modules/Leave/...` and either accept defaults or set the minimum required fields.

```php
<?php

declare(strict_types=1);

namespace Tests\Feature\Loans;

use App\Modules\Loans\Models\EmployeeLoan;
use App\Modules\Loans\Services\LoanService;
use App\Modules\Auth\Models\Role;
use App\Modules\Auth\Models\User;
use Illuminate\Foundation\Testing\RefreshDatabase;
use Tests\TestCase;

class LoanBulkApproveTest extends TestCase
{
    use RefreshDatabase;

    public function test_bulk_approve_returns_success_and_failure_buckets(): void
    {
        $approverRole = Role::firstOrCreate(['slug' => 'system_admin'], ['name' => 'System Admin']);
        $approver = User::factory()->create(['role_id' => $approverRole->id, 'is_active' => true]);

        // Reachable loan (the factory should put it in a state approve() accepts).
        $loan = EmployeeLoan::factory()->create();

        $svc = app(LoanService::class);
        $result = $svc->bulkApprove([$loan->id, 99999], $approver, 'bulk');

        $this->assertIsArray($result['approved']);
        $this->assertIsArray($result['failed']);
        $this->assertGreaterThanOrEqual(1, count($result['failed'])); // 99999 not found
    }
}
```

NOTE: This test is intentionally lenient on the "approved" count — the loan factory may produce loans in a state that `approve()` rejects (e.g. needs an active workflow). The minimum bar is: the bulk endpoint runs without throwing and returns the right shape with at least the not-found row in `failed`. If the implementer can produce a loan in approvable state without expanding scope (no extra factory creation logic), prefer asserting `count($result['approved']) === 1`.

## Acceptance Criteria

1. `POST /api/v1/leaves/requests/bulk-approve-dept` with body `{ids: [hash1, hash2], remarks?}` returns `{ data: { approved: [...], failed: [...] } }` and advances each PendingDept row to PendingHr.
2. `POST /api/v1/leaves/requests/bulk-approve-hr` similarly advances each PendingHr row to Approved.
3. `POST /api/v1/loans/bulk-approve` invokes the existing approval-chain flow per row.
4. One bad row (wrong state, not found, unauthorized) does not abort the batch — it lands in `failed[]` with a reason string.
5. Permission middleware matches single-row endpoints (`leave.approve_dept`, `leave.approve_hr`, `loans.approve`).
6. Tests pass; no NEW regressions.
